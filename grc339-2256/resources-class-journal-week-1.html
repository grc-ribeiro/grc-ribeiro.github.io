I used this link to help myself better understand the differences between CSS grid and flexbox: https://www.geeksforgeeks.org/css/comparison-between-css-grid-css-flexbox/.

Thanks for sharing, Brennan!

Flex-shrink can get tricky quickly so it may be easier to think of the flex-basis as minimum width and set the flex-shrink to 0. This allows content to grow but not shrink.

That’s my suggestion.

How do you decide when to use flexbox versus grid? Do some frontend devs tend to default to one over the other, or are there “best practices” that require avoiding one unless it’s truly necessary?

Very good question. I don’t believe that there are any “best practices”, since they’re both valid approaches. It’s a matter of personal (for an individual) or team preferences. Flexbox has become more popular because you can achieve everything you can achieve with grids, while the opposite is not true.

I learned that giving an element the value “flex” for the “display” property makes the element a flex container, not a flex item.

That’s also a very important thing to keep in mind.

I learned that giving the flex-wrap property a value of wrap will allow flex items to break into another line when necessary (if element is set to row).

You probably want that.

I was wondering if we only want to include, for example, the flex-grow and flex-shrink, could we still use the flex property as a shorthand version?

Yes. Excelent question! If you include only two values, with no units, then they will be your values for flex-grow and flex-shrink. If you include a unit in the second value, e.g. flex: 2 10em, then this shorthand will be expanded by the browser to flex-grow: 2; flex-shrink: 1; flex-basis: 30em;. You can read more on the Flex property page on the excellent MDN.

It was also useful to see how to nest selectors like section p or section a could target elements more specifically.

That’s also a very important thing to keep in mind.
